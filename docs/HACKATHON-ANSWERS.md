카투사(카카오 투자하는 사람)는 초보 투자자의 질문을 출처가 있는 비교 설명으로 바꾸고, 이해 결과에 따라 설명 방식만 조정하면서 비권유·비실행·확인 없는 성향 변경 금지를 고정하는 Codex 플러그인입니다.

카카오페이증권에서 초보 투자자용 설명 흐름을 설계·검토하는 서비스 기획자, UX 라이터, 안전·컴플라이언스 검토자를 가정 사용자로 두었습니다. 고객이 “지금 사야 하나요?”처럼 결론부터 묻지만 목표·기간·손실 감수 범위가 비어 있거나, 현재 정보의 출처가 없거나, 같은 설명을 반복해도 이해하지 못하는 지점에서 사용합니다. 플러그인은 결정을 대신하지 않고 필요한 정보와 위험을 분리해 보여 주며, 다음 확인 질문과 설명 형식 변경 후보를 남깁니다. 카카오페이증권 공식 제품이 아닌 독립 해커톤 프로토타입입니다.

## 문항 2 — 왜 이 문제를 선택했나요?

- 제한: 800자
- 현재 글자 수: **472자**

문제 인터뷰에서 초보 투자자는 매수·매도 판단 앞에서 무엇을 기준으로 결정해야 할지 막히며, 평가 초점도 결과보다 사용자가 이해하고 나아가는 과정에 있었습니다. 금융위원회 안내에서도 고객 정보, 중요 위험 설명, 이해 확인이 중요하게 다뤄집니다. 그래서 “더 그럴듯한 답”보다 “설명은 개인화해도 안전 기준은 변하지 않는가”를 문제로 골랐습니다.

기존 설명형 AI는 한 번의 답을 잘 만들 수 있지만, 사용자가 숫자·사례·체크리스트 중 무엇을 이해하기 쉬워하는지 다음 대화에 반영할 때 위험이 생깁니다. 이해 실패를 위험성향 변화로 오해하거나, 손익을 근거로 더 공격적인 설명을 하거나, 출처 없는 현재 정보를 확정적으로 말할 수 있기 때문입니다. 저는 적응 가능한 표면과 고정된 안전 척추를 분리하고, 이해 결과는 한 번에 한 필드의 변경 후보로만 기록하는 방식을 택했습니다. 실제 사용자 이해 향상과 투자 성과는 아직 검증하지 않았습니다.

## 문항 3 — 플러그인은 어떻게 작동하나요?

- 제한: 1000자
- 현재 글자 수: **657자**

플러그인은 ① 질문을 buy/sell/hold/compare/learn/unclear로 분류하고 ② 목표·기간·손실 경계·유동성·현재 정보의 출처와 관찰 시각을 확인합니다. 부족한 질문은 한 라운드 최대 3개만 묻습니다. ③ 안전 헌법을 매번 새로 읽고, 기억에서는 명시적 설명 선호와 확인된 이해 결과만 불러옵니다. ④ 사실·해석·미확인을 분리해 중립적 선택 경로와 trade-off를 설명하고 ⑤ 사용자가 손실 가능성과 판단 변경 조건을 다시 말하게 합니다. ⑥ 이해 결과로 바꿀 수 있는 것은 길이·형식·어휘·질문 밀도뿐이며, 한 번에 한 필드의 후보만 만듭니다. 목표·기간·손실 감수 범위·hard constraint는 확인 없이는 적용하지 않습니다.

결정론 코드는 output schema, 출처·timestamp, 금지된 추천·보장·긴급성 표현, one-change rule, audit/forget, 3회 repair cap을 검증합니다. 출처가 없으면 BLOCKED_NEEDS_EVIDENCE, 핵심 출처가 충돌하면 BLOCKED_HUMAN_REVIEW, 같은 오류가 3회 반복되면 중단합니다. P&L은 성향이나 안전 규칙의 근거가 될 수 없습니다. 결과는 Markdown과 JSON으로 남기며 validator가 실행되기 전에는 PASS라고 쓰지 않습니다.

## 문항 4 — AI를 어떻게 썼나요?

- 제한: 800자
- 현재 글자 수: **583자**
- 대화 기록 상태: **PRIVACY-SAFE RECONSTRUCTED CONTEXT / 직접 인용 아님**

AI에는 문제 정의를 구현 계약으로 쪼개고, 코드·fixture·unittest·replay trace·ZIP validator를 만드는 일을 맡겼습니다. 저는 설명 방식만 학습시키고 안전 규칙은 고정한다는 원칙, P&L 기반 성향 변경 금지, 한 번에 한 필드만 바꾸는 승인 구조를 직접 결정했습니다.

첫 위기는 테스트가 모두 통과한 뒤 실제 CLI 흐름에서 발견됐습니다. forget이 과거 사건을 지우자 행 개수가 줄어 기존 `evt_0002`를 다시 발급했습니다. 단순 테스트 보강으로 덮지 않고 ID를 최대 suffix+1로 바꾸고 삭제 순서가 다른 회귀 테스트까지 추가했습니다. 다음에는 AI가 ZIP 생성과 검증을 병렬 실행해 검증기가 파일보다 먼저 도착했습니다. 의존 작업 병렬화 제안은 받아들이지 않고 순차 gate로 고쳤습니다. 마지막으로 원본 로그 속 검사 명령의 변조 표식 문자열이 실제 변조로 오탐됐습니다. 로그를 지우거나 탐지를 전역 해제하는 우회는 거절하고, JSONL의 `item.command`만 예외로 두되 다른 필드의 표시는 계속 차단했습니다. 이 과정을 29개 테스트와 최종 ZIP 검증으로 고정했습니다.

## 문항 5 — 어떻게 검증했나요?

- 제한: 800자
- 현재 글자 수: **682자**

대표 정상 입력은 “ETF 적립식 매수와 예금 유지 차이를 알고 싶다”였습니다. 출처·관찰 시각·손실 가능성을 포함하자 READY_FOR_USER_DECISION/PASS가 나왔습니다. 출처 없는 현재 정보는 BLOCKED_NEEDS_EVIDENCE, 보장 수익 표현은 validator FAIL, 확인 없는 material 변경과 P&L 기반 변경은 차단했습니다. 이해 실패 뒤 숫자형을 요청한 사례에서는 explanation_style.format 하나만 후보가 되고 사용자 확인 전에는 적용되지 않는지 확인했습니다.

unittest 29개, 10개 사례×3개 성향 30회 replay, 안전 불변식 불일치 0건, plugin/skill validator 통과를 재현했습니다. CLI smoke에서 checklist→numbers 적용 시 profile v2가 되었고, forget 후 원문 payload 0건과 deletion receipt만 남았습니다. 테스트 중 forget 후 event ID 중복 가능성을 의심해 max suffix 방식과 회귀 테스트 2개를 추가했습니다. 개인정보를 제거하고 실제 사건 순서만 보존한 재구성 개발 맥락을 포함한 FINAL ZIP은 레이아웃·SHA-256·비밀 검사를 통과했습니다. 실제 사용자의 이해·성과, 법률·컴플라이언스 적합성, 실시간 데이터는 검증하지 않았습니다.
